# 产品参考与差异化

OpenAdOps 不复制单一广告软件，而是组合参考专业产品中最有价值的工作方式。

| 参考产品 | 借鉴能力 | OpenAdOps 的差异 |
| --- | --- | --- |
| [Smartly](https://www.smartly.io/) | 创意与媒体工作流联动、批量搭建、跨媒体管理 | 从不完整客户资料开始，先形成代理商可辩护的决策链路 |
| [Motion](https://motionapp.com/) | 素材归类、创意假设、表现解释、下一条素材方向 | 把素材假设放入 Offer、策略、上线和复盘的完整上下文 |
| [Motion MCP](https://help.motionapp.com/en/articles/14315735-motion-mcp) | 以只读 MCP 将真实广告数据交给 AI 工具 | 保持本地优先，并为未来的只读媒体 Connector 预留接口 |
| [Optmyzr](https://www.optmyzr.com/) | 优化建议、预算监控、规则、护栏与审批 | 先服务 Google、Meta、TikTok 的代理商工作流，不直接写广告账户 |
| [Madgicx](https://madgicx.com/) | AI Media Buyer 式的下一步行动建议 | 不局限 Meta，强调证据、假设和人工确认 |
| Google / Meta / TikTok 原生后台 | 最终 Campaign 设置和媒体执行 | OpenAdOps 是决策与交付层，不重新制造 Ads Manager |
| [Google Ads Experiment Center](https://support.google.com/google-ads/answer/16856494) | 随机分流、实验状态、原生结果和应用赢家 | 在进入媒体后台前冻结假设、样本门槛、护栏和证据 |
| [Meta Ads Manager A/B test](https://www.facebook.com/help/messenger-app/621956575422138/) | 在 Campaign 创建流程中启用 A/B test | 统一记录跨行业、跨媒体的实验逻辑和学习结论 |
| [TikTok Split Testing](https://ads.tiktok.com/help/article/split-testing) | 互斥受众分组与 90% confidence 的原生判断 | 不把 TikTok 的 90% 规则错误套用到其他媒体 |

## 产品机会

现有工具通常从“广告账户已经存在”或“素材数据已经存在”开始。代理商更早的工作却常常是：

- 客户只发了一段 Offer，信息不完整。
- 客户可能给出一段不可直接执行的策略。
- 优化师需要补齐问题、判断媒体分工、派素材并搭建广告。
- 上线前缺少一个能统一检查策略、追踪、合规和执行风险的交付物。

OpenAdOps 的核心机会是把这段前期工作产品化，并让后续数据复盘回流到同一个项目。

## 借鉴边界

- 借鉴工作流和信息架构，不复制视觉、文案或封闭平台实现。
- 专业规则需要注明适用条件，不能把行业参考值当作客户事实。
- 涉及平台政策、接口和追踪规范的规则必须允许更新和替换。
